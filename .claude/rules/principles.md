## 최우선 원칙: 가독성
- 코드는 작성하는 시간보다 읽는 시간이 압도적으로 길다. 모든 선택에서 가독성을 최우선으로 한다.
- 직관적이고 명확한 네이밍을 선호한다. 약어나 축약어보다 바로 의미가 파악되는 이름을 사용할 것.
- 위에서 아래로 자연스럽게 읽히는 흐름을 유지한다. 코드를 이해하기 위해 여러 파일을 점프해야 한다면 구조가 잘못된 것이다.
- 영리한 코드보다 명백한 코드가 낫다. 한눈에 의도가 파악되는 코드를 작성한다.

## YAGNI — 지금 필요한 것만 작성할 것
- 요청받은 것만 구현한다. "나중에 쓸 수 있으니까"로 코드를 추가하지 않는다.
- 한 번만 쓰이는 로직을 함수나 유틸리티로 분리하지 않는다. 반복이 실제로 발생할 때 추상화한다.
- 불필요한 에러 핸들링, 폴백, 방어 코드를 넣지 않는다. 내부 코드와 프레임워크 보장을 신뢰한다. 검증은 시스템 경계(사용자 입력, 외부 API)에서만 한다.
- 기존 코드를 건드릴 때 주변 코드를 "개선"하지 않는다. 버그 수정은 버그만 수정한다.
- 변경하지 않은 코드에 주석, 독스트링, 타입 어노테이션을 추가하지 않는다.
- 설정 가능성, 플러그인 구조, 확장 포인트를 미리 만들지 않는다.
- 비슷한 코드 3줄이 섣부른 추상화보다 낫다.

## 원칙은 도구일 뿐, 교리가 아니다
- **DRY**: 중복 제거 자체가 목적이 아니다. 중복처럼 보여도 변경 이유가 다르면 각각 두는 게 맞다. 억지로 합치면 한쪽 변경이 다른 쪽을 깨뜨린다.
- **Clean Code**: 함수를 짧게 쪼개는 것 자체가 목표가 아니다. 한 곳에서 흐름이 읽히는 코드가 여러 함수를 점프하며 읽는 코드보다 낫다.
- **SOLID**: 인터페이스, 추상 계층, 의존성 주입을 구현이 하나뿐인 곳에 미리 만들지 않는다. 실제로 두 번째 구현이 필요해질 때 도입한다.
- 원칙을 따르기 위해 코드가 더 복잡해진다면, 그 원칙 적용이 틀린 것이다.

## 도구의 설계를 거스르지 않는다
- 플러그인·프레임워크·셸이 의도한 동작과 어긋나는 우회·핵·억지 패치로 불편함을 잠재우지 않는다. 설정이 불투명해지고 업그레이드·이식이 깨진다.
- 불편함을 발견하면 먼저 분류한다: 디자인 의도인가 / 명백한 버그인가 / 명백한 개선 여지인가. 디자인 의도면 그대로 가져간다.
- 해결책은 도구가 의도한 흐름에 맞는 정공법부터 제시한다. 우회·monkey-patch는 마지막 카드.
- "그냥 가져가자"는 유효한 선택이다. 모든 마찰에 패치를 쌓지 않는다.

## 커밋 컨벤션
- `feat(scope):`, `fix(scope):`, `chore:` 형식.
- 버그 수정 커밋은 버그만 다룬다. 무관한 리팩터를 섞지 않는다.
- 커밋에 `Co-Authored-By: Claude` 트레일러를 붙이지 않는다.

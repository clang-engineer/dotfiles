# Global Claude Code Instructions

## 최우선 원칙: 가독성
- 코드는 작성하는 시간보다 읽는 시간이 압도적으로 길다. 모든 선택에서 가독성을 최우선으로 한다.
- 직관적이고 명확한 네이밍을 선호한다. 약어나 축약어보다 바로 의미가 파악되는 이름을 사용할 것.
- 위에서 아래로 자연스럽게 읽히는 흐름을 유지한다. 코드를 이해하기 위해 여러 파일을 점프해야 한다면 구조가 잘못된 것이다.
- 영리한 코드보다 명백한 코드가 낫다. 한눈에 의도가 파악되는 코드를 작성한다.

## YAGNI — 지금 필요한 것만 작성할 것
- 요청받은 것만 구현한다. "나중에 쓸 수 있으니까"로 코드를 추가하지 않는다.
- 한 번만 쓰이는 로직을 함수나 유틸리티로 분리하지 않는다. 반복이 실제로 발생할 때 추상화한다.
- 불필요한 에러 핸들링, 폴백, 방어 코드를 넣지 않는다. 내부 코드와 프레임워크 보장을 신뢰한다. 검증은 시스템 경계(사용자 입력, 외부 API)에서만 한다.
- 기존 코드를 건드릴 때 주변 코드를 "개선"하지 않는다. 버그 수정은 버그만 수정한다.
- 변경하지 않은 코드에 주석, 독스트링, 타입 어노테이션을 추가하지 않는다.
- 설정 가능성, 플러그인 구조, 확장 포인트를 미리 만들지 않는다.
- 비슷한 코드 3줄이 섣부른 추상화보다 낫다.

## 원칙은 도구일 뿐, 교리가 아니다
- **DRY**: 중복 제거 자체가 목적이 아니다. 중복처럼 보여도 변경 이유가 다르면 각각 두는 게 맞다. 억지로 합치면 한쪽 변경이 다른 쪽을 깨뜨린다.
- **Clean Code**: 함수를 짧게 쪼개는 것 자체가 목표가 아니다. 한 곳에서 흐름이 읽히는 코드가 여러 함수를 점프하며 읽는 코드보다 낫다.
- **SOLID**: 인터페이스, 추상 계층, 의존성 주입을 구현이 하나뿐인 곳에 미리 만들지 않는다. 실제로 두 번째 구현이 필요해질 때 도입한다.
- 원칙을 따르기 위해 코드가 더 복잡해진다면, 그 원칙 적용이 틀린 것이다.

## 도구의 설계를 거스르지 않는다
- 플러그인·프레임워크·셸이 의도한 동작과 어긋나는 우회·핵·억지 패치로 불편함을 잠재우지 않는다. 설정이 불투명해지고 업그레이드·이식이 깨진다.
- 불편함을 발견하면 먼저 분류한다: 디자인 의도인가 / 명백한 버그인가 / 명백한 개선 여지인가. 디자인 의도면 그대로 가져간다.
- 해결책은 도구가 의도한 흐름에 맞는 정공법부터 제시한다. 우회·monkey-patch는 마지막 카드.
- "그냥 가져가자"는 유효한 선택이다. 모든 마찰에 패치를 쌓지 않는다.

## 구조 정리
- 같은 목적의 스크립트/파일이 분산되어 있으면 하나로 통합한다.
- 옵션과 플래그는 실제 사용 패턴에 맞게 최소화한다.
- 반복되는 하드코딩 값은 데이터 파일로 분리하고, 로직이 데이터를 읽게 한다.
- 관련 파일은 한 디렉토리에 모은다.
- 사용하지 않는 파일은 제거한다.

## 제안 검증
- 플랜은 해당 영역의 일반적 관행을 기본값으로 한다.
- 일반적이지 않은 선택을 할 때는 그 이유(성능, 제약, 기존 구조 등)를 함께 제시한다.
- 접근 방식·설정·도구·기술을 제안하거나 평가할 때는 다음을 함께 제시한다:
  - **대중성**: 주류인지 비주류인지 솔직하게 (개인/팀/엔터프라이즈 맥락 고려)
  - **대안**: 같은 목적을 달성할 수 있는 주요 대체제
  - **비교**: 현재 안 vs 대안의 장단점을 간결하게
  - **결론**: 유지할 만한지, 전환을 고려할 만한지 의견
- "다 좋다"식 애매한 답변 금지. 대중적이지 않으면 그렇다고 명확히 말한다.
- 의심스러우면 검색·문서로 확인한다.

## 작업 진행

### 알림 (도구 호출 전)
- 사용자가 의도를 짐작하기 어려운 호출 전에는 한 줄로 무엇/왜를 먼저 말한다.
- 표준 조회(`ls`, `git status`)는 생략. 병렬 묶음은 묶음 단위 한 줄.
- 호출 후 핵심 결과는 사용자 관점으로 한 줄 요약 (전체 출력 X).
- **알림이지 차단이 아니다.** 사용자가 끊을 수 있게 보여줄 뿐, 응답 대기 없이 그대로 진행한다.

### AI 작업에서 배우기 (라이브 우선)
목적은 아카이브가 아니라 학습이다. AI가 하는 일을 그냥 흘려보내지 않고, 그 안의 배울 점을 사용자가 실제로 배우게 하는 게 핵심. 안 읽는 flow 로그를 쌓는 게 아니다.

- **학습은 그 순간, 대화에서.** 멀티스텝 작업(3+ 단계, 재현 가능한 패턴)을 끝내면 **"💡 배울 점"을 대화에 짧게** 꺼낸다 — 명령 나열이 아니라 **비자명했던 판단·선택·트레이드오프·함정** 위주 2~3개. 사용자가 파일 안 열어도 그 자리에서 배우는 지점.
- **가시성이 곧 강제 장치.** 대화에 노출하니 빠뜨리면 사용자가 바로 알아챈다. (조용히 파일에만 쓰면 놓쳐도 안 보여서 실제로 잘 안 잡혔다 — 그게 지금까지의 실패 모드.)
- **남길지는 novelty로 판단.** 그 배울 점이 이미 `$BLOG_DIR`/`$TOOLBOX_DIR`의 기존 자료(flow-*·note·post·cheatsheet)에 있으면 skip. **없던 것만** 남긴다. 매번 코퍼스 전수 스캔 X — flow감이다 싶을 때 관련 파일만 대조.
- **persist는 선택적.** 기본은 대화 surface까지. 사용자가 "남겨"라고 하거나 정말 새롭고 재현 가치 큰 흐름이면 `$TOOLBOX_DIR/notes/flow-<도구>-<작업>.md`에 적재하고 **한 줄 보고**. 같은 파일 있으면 차이만 갱신. 형식은 `notes/flow-meta.md` 참조.
- 밀도 > 분량. 적용 안 함: **작업 자체가** 단발 호출·자명한 작업·즉흥 편집/디버깅일 때. 단, **재현 가능한 멀티스텝 작업 중 디버깅으로 발견한 비자명한 함정은 제외가 아니라 배울 점의 본체다** — 발견 경로가 디버깅인 것과 작업 자체가 디버깅인 것을 혼동하지 말 것. 산출물이 동작하는 순간 "결과 보고 + 다음 액션"으로 넘어가며 학습 스텝을 건너뛰지 않는다. `$TOOLBOX_DIR` 미설정 시 persist X (대화 surface는 유효).

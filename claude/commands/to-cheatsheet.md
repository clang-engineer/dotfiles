현재 대화에서 다룬 명령어/도구/기술을 주제별로 정리해서
$TOOLBOX_DIR/cheatsheets 디렉토리에 마크다운 치트시트 파일로 작성해줘.

만약 $TOOLBOX_DIR 환경변수가 설정되어 있지 않다면, 치트시트를 작성하지 말고
사용자에게 다음과 같이 안내해줘:
"`TOOLBOX_DIR` 환경변수가 설정되지 않았습니다. `.bashrc` 또는 `.zshrc`에
`export TOOLBOX_DIR=~/your-toolbox-repo` 를 추가한 뒤 셸을 재시작해주세요."

## 치트시트의 정의 (가장 중요)

치트시트는 **"명령어/사용법이 생각이 안 날 때 빠르게 찾아보는 용도"**.
판단 기준은 단 하나 — **"나중에 다시 찾아볼 명령어/문법인가?"**

YES → 작성. 형식은 자유 (표/코드블록/짧은 bullet 모두 OK)
NO → 작성하지 말고 `/to-blog`를 권하기.

## 작성 금지 (자꾸 섞이는 문서적 성격)

다음은 치트시트에 넣지 말 것. 나오면 별도 글로 분리하라고 사용자에게 안내:
- 개념 설명·동작 원리 (예: "이 명령어는 내부적으로 ~를 호출한다")
- 튜토리얼·하우투 ("먼저 X를 하고, 그다음 Y를 한다" 식의 흐름)
- 배경·역사·디자인 철학
- 한 번 읽고 끝낼 가이드성 문장
- AI 도구 작업 흐름·단계 (자동으로 `notes/flow-*.md`에 적재됨)

위 내용이 대화에 있었다면: "이 부분은 `/to-blog`로 정리하는 게 좋습니다"라고 안내.

## 작성 전 확인사항

- 기존 치트시트 목록을 확인하고, 같은 주제의 파일이 이미 있으면 새로 만들지 말고 기존 파일에 내용을 추가/업데이트
- 대화 내용 중 치트시트로 정리할 만한 주제(= "다시 찾을 명령어")가 없으면 사용자에게 알려줘

## 요구사항

- 파일명: 주제명.md (소문자, 하이픈 구분. 예: docker.md, sed-awk.md)
- 제목: `# 주제 Cheatsheet` 형식
- 섹션별로 `##`으로 구분
- 명령어/옵션은 테이블(`| 명령어 | 설명 |`) 또는 코드블록으로 정리
- 각 항목은 한눈에 스캔 가능해야 함 — 설명은 한 줄로 압축
- 한국어로 작성
- 실용적인 예시 포함 (대화에서 실제 사용한 명령 우선)

추가 인자가 있으면 주제/범위 힌트로 사용: $ARGUMENTS

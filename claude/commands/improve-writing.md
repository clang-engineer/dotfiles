인자로 받은 경로($ARGUMENTS)의 **이미 게시된 글들**을 외부 독자 기준으로 품질을 끌어올린다.
내용 개선이 본체이고, 필요하면 글 단위로 병합·분할·삭제·제목/파일명 변경까지 한다.

경로 미지정 시 `$BLOG_DIR/_posts` 전체를 대상으로 본다. `$BLOG_DIR` 미설정 + 경로도 없으면
작업하지 말고 안내: "`BLOG_DIR` 미설정. 개선할 경로를 인자로 지정하거나 `export BLOG_DIR=...`."

## 이 명령의 자리

- `/to-blog` — 새 글을 **게시**할 때 (대화 → 새 포스트)
- **이 명령** — 이미 **게시된 글들**을 사후에 **개선**할 때 (기존 글 → 더 나은 글)

목적은 단 하나: **검색으로 들어온 외부 독자가 얻는 가치를 높인다.** 개인 메모 품질이 아니라
공개 글 품질 기준으로 본다.

## 게시된 글이라 특별히 챙길 것 (URL 보존)

블로그는 `jekyll-redirect-from`을 쓴다. 파일명·제목·구조를 바꾸면 URL이 깨질 수 있으므로:

- **파일명/슬러그 변경, 병합, 삭제** → 사라지는 URL은 `redirect_from`으로 살린다.
  - 살아남는(또는 흡수한) 글 front matter에 옛 경로 추가:
    ```yaml
    redirect_from:
      - /posts/old-slug/
    ```
  - 옛 URL은 permalink 규칙에서 역산한다 (`_config.yml`의 permalink 확인).
- 내용을 손대면 front matter `updated` 를 오늘 날짜로 갱신한다 (`date`는 원래 게시일 유지).
- `categories`/`tags`는 분류가 실제로 바뀔 때만 손댄다.

경로가 `$BLOG_DIR` 밖의 일반 글 디렉토리면 위 Jekyll 규칙은 건너뛰고 내용·구조 개선만 한다.

## 작업 흐름 (이 순서를 지킨다)

### 1단계: 사전 점검

- 대상 경로를 스캔해 각 글의 주제·길이·게시일을 파악한다.
- 다음을 식별한다:
  - **파편화** — 같은 주제가 여러 글에 흩어져 한 편의 완결글이 더 나은 경우
  - **중복** — 내용이 겹쳐 한쪽이 다른 쪽에 흡수되어야 하는 경우
  - **과적재** — 한 글이 독립된 주제를 2개 이상 담아 독자가 길을 잃는 경우
  - **품질 저하** — 훅 없는 도입부, 무너진 구조, 군더더기, 모호함, 깨진 예시

### 2단계: 개선 계획표 (승인용)

각 글에 대해 액션을 정하고 표로 제시한다:

```
| 파일 | 액션 | 이유 | 핵심 개선점 | URL 영향 |
```

액션은 다음 중 하나:
- **IMPROVE** — 구조는 두고 내용 품질만 개선 (목차 정리 / 부실 보강 / 군더더기 삭제 중 무엇인지 표시)
- **MERGE** — 파편/중복을 한 글로 통합 (흡수된 글은 redirect_from으로 보존 또는 삭제)
- **SPLIT** — 과적재 글을 독립 주제별로 분할
- **DELETE** — 가치 낮음/대체됨 (대체 글로 redirect_from)
- **RENAME** — 제목·파일명 개선 (검색 친화 + 내용 일치, redirect_from 필수)
- **KEEP** — 손댈 필요 없음

### 3단계: 사용자 승인 (필수 — 이 단계 없이 비가역 작업 금지)

병합·분할·삭제·파일명 변경은 되돌리기 어렵다. 계획표를 보여주고 받는다:
- 계획표가 맞는지
- 애매한 판단 1~3개를 짚어 의견 받기 (특히 DELETE/MERGE 대상)
- 일괄 vs 단계별 진행 방식

### 4단계: 실행 — 내용 품질 기준

각 글에 다음을 적용한다 (이게 본체):

- **도입부 훅** — 첫 1~2문단에서 "이 글이 무슨 문제를 풀어주는지"가 바로 잡히게
- **흐름** — 위에서 아래로 자연스럽게 읽히게. 헤딩 위계 정리, 점프 없는 논리 전개
- **스캔 가능성** — 긴 글은 소제목·목록·코드블록으로 훑어 읽을 수 있게
- **군더더기 제거** — 반복·동어반복·의미 없는 수식 삭제. 모호한 문장은 구체화
- **정확성** — 코드/명령/예시가 실제로 동작·재현되는지. 사실관계 확인
- **front matter** — `description`은 검색 스니펫 품질로 (50~100자, 핵심 한 줄). `title`은 내용과 일치하고 검색 친화적으로
- **원저자 보존** — 사실을 날조하지 않는다. 글의 목소리·관점은 유지하고 표현만 다듬는다

구조 작업(MERGE/SPLIT/RENAME/DELETE) 시 위 **URL 보존** 규칙을 반드시 함께 적용한다.

### 5단계: 보고

- 블로그 repo `git status -s` 를 보여준다.
- 자동 커밋하지 않는다 — staged 상태로 두고 사용자가 검토 후 커밋.
- 요약: 글별 액션 분포, 개선 핵심, redirect 추가한 URL 목록, 발견했지만 안 건드린 이슈.

## 절대 하지 말 것

- 계획표 승인 없이 병합/분할/삭제/파일명 변경
- 게시된 글의 URL을 redirect_from 없이 깨뜨리기
- 내용 날조 또는 원저자 관점 임의 변경
- 자동 git commit

추가 인자는 범위 힌트로 쓴다: $ARGUMENTS
(예: "vim 카테고리만" "삭제는 빼고" "내용 개선만, 구조는 두고" 등)

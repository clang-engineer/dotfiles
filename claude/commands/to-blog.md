현재 대화에서 다룬 **공개 가치 있는 내용**을 정리해서
$BLOG_DIR 환경변수로 지정된 블로그 프로젝트의 _posts 디렉토리에
Jekyll 블로그 포스트 형식의 md 파일로 작성해줘.

만약 $BLOG_DIR 환경변수가 설정되어 있지 않다면, 포스트를 작성하지 말고
사용자에게 다음과 같이 안내해줘:
"`BLOG_DIR` 환경변수가 설정되지 않았습니다. `.bashrc` 또는 `.zshrc`에
`export BLOG_DIR=~/your-blog-repo` 를 추가한 뒤 셸을 재시작해주세요."

## 블로그의 정의

**"검색해서 들어온 외부 독자가 가치를 얻을 만한 모든 완성글"**.
형식은 제한 없음 — 다음 중 어떤 것이든 OK:
- **트러블슈팅 글** — 이슈/원인/해결 기록
- **가이드·튜토리얼** — 개념·문법·사용법 정리 (예: "Lua 문법 시리즈", "Vim 옵션 가이드")
- **학습 정리** — 새로 배운 기술의 핵심 정리
- **하우투** — 특정 작업 단계별 가이드

판단 기준: 외부 독자가 검색으로 들어와 가치를 얻을 만한가?

## 다른 명령어와 구분

- **짧은 개인 학습 메모** (한 줄 발견) → `/to-til`
- **다시 찾을 명령어/문법** (빠른 참조용 스니펫) → `/to-cheatsheet`
- **특정 프로젝트/코드베이스 분석** (비공개) → `/to-analysis`
- **그 외 공개 가치 있는 글** → 이 명령어

cheatsheet와 가르는 기준:
- "다시 찾을 명령어 표"는 cheatsheet
- "왜 그렇게 동작하는지, 어떻게 쓰는지" 설명이 메인이면 blog
- 같은 주제(예: vim 옵션)를 다뤄도 보완 관계 — cheatsheet는 빠른 참조, blog는 가이드

## 작성 전 유사 글 체크

- `$BLOG_DIR/_posts`에서 제목 키워드/주제가 겹치는 기존 글을 먼저 검색한다.
- 발견되면 새 글을 쓰지 말고 사용자에게 보고: 유사 글 경로/제목을 보여주고
  "기존 글에 추가 vs 새 글 작성" 중 어떻게 할지 묻는다.
- 기존 글에 추가하기로 한 경우, 본문 끝에 `## YYYY-MM-DD 추가` 섹션을 만들어
  새 내용만 덧붙인다(기존 발행일/내용은 건드리지 않는다).
- 이미 다뤄진 내용이 그대로 포스팅되지 않도록, 추가 전에 중복 구간을 확인하고
  새로 알게 된 부분만 정리해서 쓴다.

## 요구사항

- 파일명: YYYY-MM-DD-제목.md (오늘 날짜 사용)
- Front matter 포함 (layout, title, date, categories, tags)
  - categories는 `[디렉토리, "서브카테고리"]` 2단계 형식 (디렉토리별 서브카테고리는 사용자 메모 참고)
  - **가이드성 글**(개념·문법·튜토리얼·종합 정리)이면 `tags`에 `guide` 추가 — `/tags/guide/` 모음 페이지에 노출됨
- 글 유형에 맞게 구조화:
  - 트러블슈팅: 이슈 상황 → 원인 분석 → 해결 방법
  - 가이드·튜토리얼: 개념 → 사용법 → 예시
  - 학습 정리: 핵심 개념 → 예시 → 정리
- **외부 독자 톤**: 개인 맥락은 일반화, 사적 메모 톤(축약·반말) 다듬기
- 코드 블록에 언어 지정
- 한국어로 작성

추가 인자가 있으면 제목/카테고리 힌트로 사용: $ARGUMENTS
